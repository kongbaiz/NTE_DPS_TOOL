# NTE DPS Meter 关键功能与数据行为分析

分析日期：2026-06-10  
目标程序：`NTE_DPS_Meter.exe`  
SHA-256：`6B4871ED4E8B276B0AE3250CED4B9A727C6347C121C25C718BA86DADC54FCFA7`

配套更新器：`NTE_Updater.exe`  
SHA-256：`0E6F87C04389896E166E5C8D588231A8966698012E43AAE96165BAFA9EC63A7A`

## 一、结论摘要

1. 伤害、命中、角色识别和战斗阶段数据来自游戏网络封包解析，不是通过读取游戏进程内存取得。
2. 未发现 `OpenProcess`、`ReadProcessMemory`、`WriteProcessMemory`、`NtReadVirtualMemory`、远程线程注入等进程内存访问链。
3. 程序存在战报统计上传功能，会发送设备标识、Discord 标识、授权令牌、角色阵容、伤害、命中和战斗场景等数据。
4. 战报上传位于“战报管理”界面，需要用户选择战报并在模态对话框中勾选同意；当前版本没有发现后台自动上传战报的调用链。
5. 授权系统是独立的自动外联面：激活、自动续验以及程序退出时的反激活均可能访问授权服务器，并使用持久化设备 ID。
6. 程序会枚举游戏进程、TCP 连接、路由和网络接口，以选择抓包网卡及判断代理、VPN、加速器环境。这是系统网络诊断，不是读取游戏内存。
7. 启动公告会访问远程更新元数据，并能够启动同目录下的独立更新器。
8. 客户端内嵌固定统计 API 密钥。报告中不公开完整密钥，但该设计不能构成真正的服务端鉴权，应立即轮换并改造。

## 二、伤害统计主链

核心模块包括：

- `engine.sniff_engine`：选择网卡、抓取游戏流量、TCP 流重组。
- `engine.s2c_parser`：解析服务端到客户端的状态、场景和会话数据。
- `engine.core`：提取命中记录、累计伤害、DPS、角色统计并保存战报。
- `engine.session_mode`：识别不同副本、Boss、阶段和多目标战斗。
- `engine.report`：读取本地战报。
- `engine.bridge`：以约 33 ms 周期向 UI 推送统计快照，并进行约 200 ms 更新节流。

`engine.session_mode` 内置多种模式处理器：

- `advvision`
- `weekly`
- `anomaly_hunt`
- `abyss`
- `free`

它通过封包中的场景标记、对象生成/退出事件、F1/F2 数值和 Boss 最大生命值表识别战斗阶段。可生成开始、保存并结束、保存并开启下一阶段等状态转换。

本地统计能力包括：

- 总伤害、实时 DPS、命中次数。
- 按角色和技能分类的伤害统计。
- 第一击至最后一击的有效战斗时长。
- 多目标和分阶段 Boss 的剩余时间估算。
- 最多约 50,000 条命中历史。
- 最多约 10,000 个诊断封包缓存。
- 将战报保存到 `workspace/reports`。
- 将诊断封包以十六进制形式导出到 `workspace/captures/diag_*.json`。

## 三、统计上传

服务地址：

`https://nte-dps-meter-analytics.vercel.app/api/upload`

请求方式：

- `POST`
- JSON 请求体
- 请求头包含固定的 `x-api-key`
- 超时约 15 秒
- 后台守护线程逐笔上传

上传 payload 可包含：

- `device_id`
- `discord_id`
- `analytics_token`
- 程序版本
- 是否为模拟数据
- 战报时间戳
- 有效战斗时长
- 总伤害及分级
- 角色 ID、角色名、角色伤害、命中数
- 队伍阵容
- 战斗模式
- Boss、阶段、路线、站点、上下半场等会话信息

上传资格中存在最低总伤害阈值 `50,000`。战斗时长还会被映射为不同 tier。

### 触发条件

已确认的调用链为：

`ReportManageDialog._upload_selected`
→ 从磁盘加载被勾选的战报
→ `build_upload_payload`
→ `UploadConsentDialog`
→ 用户勾选同意并确认
→ 后台线程调用 `upload_one`

上传模块自身的说明明确表示“由 ReportManageDialog 调用，不自动触发”。同意框的上传按钮初始为禁用状态，只有勾选同意后才启用。

因此，对当前样本的准确表述应是：

> 存在包含设备和玩家关联标识的战报上传能力，但未发现程序启动后或每场战斗结束后自动上传战报的证据。

### 风险

- 固定 API 密钥随客户端发布，任何人都可以从二进制中恢复，不能作为可信鉴权凭据。
- 战报与持久设备 ID、Discord ID 组合后可以形成稳定的玩家画像。
- 战斗模式、阵容、时间戳及详细伤害记录超出了实现本地 DPS 显示所必需的数据范围。
- 用户同意是否充分取决于界面条款是否明确逐项披露上述字段、保存期限、用途和删除方式。

## 四、授权与设备标识

授权服务：

`https://nte-dps-meter-license.vercel.app`

接口：

- `/api/activate`
- `/api/deactivate`

授权状态保存在 `settings.json`，包括：

- 序列号状态
- 剩余天数和到期时间
- `analytics_token`
- `device_id`
- `discord_id`
- `discord_name`

如果不存在设备 ID，程序会生成 UUIDv4 并持久化。授权状态机的模块说明为：

`offline -> gaming (activate) -> offline (deactivate / 程序关闭)`

程序还注册了退出处理函数，可能在退出时调用反激活接口。授权请求超时约 10 秒。

免费功能：

- 全局热键
- 实时 DPS
- 悬浮窗
- 命中日志

付费功能：

- 战斗报告
- 报告比较
- 导出

### 风险

- 设备 ID 属于可持续关联同一安装实例的标识，应在隐私说明中披露。
- 授权验证和统计分析共用 `analytics_token`、设备 ID、Discord 信息，存在跨功能关联风险。
- 应说明反激活请求在何时自动发生、发送哪些字段，以及网络失败时如何处理。

## 五、Discord 绑定

绑定流程会打开浏览器访问：

`https://nte-dps-meter-analytics.vercel.app/discord-link?device_id=<设备标识>`

随后客户端约每 3 秒轮询：

`/api/discord-status?device_id=<设备标识>`

最长轮询时间约 1,800 秒，单次请求超时约 10 秒。绑定成功后保存：

- `discord_id`
- `discord_name`

IDA 对 `DiscordLinkWorker.run` 的反编译地址为 `0x1400C24E0`，确认了浏览器跳转和轮询流程。

## 六、网络与进程探测

`engine.net_detect` 使用 Windows 系统接口执行：

- 枚举 IPv4/IPv6 TCP 连接。
- 通过 `GetExtendedTcpTable` 获取连接所属 PID。
- 通过进程快照获取可执行文件名。
- 查找 `HTGame.exe`。
- 以 TCP 端口 `30031` 作为游戏连接锚点之一。
- 枚举路由、网卡和接口地址。
- 判断直连、系统代理、VPN 或游戏加速器环境。

程序内含 UU、GearUp、雷神、迅游等加速器进程特征，以及 WireGuard、OpenVPN、Tailscale、ZeroTier、Hyper-V 等接口特征。

诊断导出可能包含：

- 进程名称和 PID
- 游戏服务器 IP
- 本地/远端端口和 TCP 连接
- 网卡、路由、VPN/代理判断

这些能力解释了程序为什么需要查看系统进程和网络状态，但没有证据表明其读取了 `HTGame.exe` 的地址空间。

## 七、启动公告与更新

启动公告模块会请求：

`https://nte-dps-meter-update.vercel.app/latest.json`

请求使用包含程序版本的 User-Agent，超时约 5 秒。失败时使用内置公告内容。

公告界面能够通过 `subprocess.Popen` 启动同目录下的：

`NTE_Updater.exe`

当前分析确认了主程序的远程版本元数据请求和更新器启动链，但没有完成更新器内部下载完整性与签名验证逻辑的逐函数审计。因此不能据此断言更新包是否具有哈希校验、数字签名校验或回滚保护。

## 八、全局热键和窗口探测

热键模块使用 Windows `RegisterHotKey`，在独立线程中接收已注册组合键：

- `Alt+D`：悬浮窗
- `Alt+E`：主窗口
- `F6`：重置

这不是键盘记录器：已发现的是有限组合键注册，没有发现捕获所有键盘输入的低级键盘钩子。

主窗口还会：

- 通过 `tasklist` 判断 `HTGame.exe` 是否运行。
- 枚举可见窗口并取得游戏窗口矩形。
- 根据游戏窗口位置调整悬浮窗。

## 九、其他业务功能

除基础 DPS 外，程序还包含：

- 战报筛选、批量删除和批量上传。
- 战报对比。
- 角色/技能排行。
- 队伍搭配或配对页面。
- 角色养成信息。
- ECG 风格实时曲线面板。
- 多语言切换。
- 悬浮窗和游戏窗口跟随。
- 模拟数据模式。
- 分阶段、副本和多目标战斗识别。

## 十一、建议向官方提出的整改项

1. 公布完整数据清单，逐项说明本地采集、网络发送、用途、保存期限、共享对象和删除渠道。
2. 将授权请求、Discord 绑定、公告更新、战报上传分别提供独立的网络行为说明。
3. 战报上传前展示实际发送字段预览，并允许用户移除设备 ID、Discord ID、时间戳和详细命中记录。
4. 默认不上传；不得以付费功能或绑定流程诱导用户同意非必要统计。
5. 为用户提供撤回同意、解除设备与 Discord 关联、删除历史上传数据的入口。
6. 立即轮换客户端中暴露的固定 API 密钥，改用服务端签发的短期、限权限令牌，并实施速率限制和重放防护。
7. 将授权身份与社区统计身份分离，避免使用相同设备 ID 和令牌形成跨业务画像。
8. 对更新包实施代码签名、签名链校验、版本防回滚和发布清单校验，并公开更新器的验证机制。
9. 对诊断文件明确提示其可能包含服务器 IP、进程名、PID、网卡和路由信息，导出前进行脱敏。
10. 提供可复现的协议和算法说明，或提交源代码供游戏官方安全审计。

## 十二、分析边界

本报告基于指定二进制的 IDA 静态分析、Nuitka 模块恢复和已有网络抓包交叉验证。静态分析能够证明代码能力和主要调用关系，但不能单独证明服务器实际保存、分享或二次使用数据的方式。相关服务器日志、数据库结构、隐私政策和删除接口仍需由运营方或游戏官方进一步核验。
